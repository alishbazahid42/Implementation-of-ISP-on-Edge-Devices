What is SYEISPNet?

Image Signal Processor (ISP) Network

Replaces the traditional camera ISP pipeline in software. Takes a 4-channel 16-bit RAW sensor image (RGGB Bayer mosaic) and converts it to a 3-channel sRGB image, trained to match Fujifilm color science.


Two variants exist: SYEISPNet (training, 110K params, multi-branch) and SYEISPNetS (slim, 5,640 params, single conv per block — used at runtime).

Why Two Variants?

Multi-branch training gives the optimizer more gradient paths → finds better weights → higher PSNR. But at inference, only one equivalent conv is needed — the branches exist purely for training.


Reparameterization mathematically fuses all 10 branches into one. Output is bit-identical (PSNR = 134 dB between multi-branch and slim). Zero quality loss, 20× fewer parameters.

Dataset
2,417 paired images captured simultaneously: MediaTek smartphone RAW sensor as input, Fujifilm camera as the "ideal ISP" reference output.

Input: MediaTek RAW (mediatek_raw/)

  • Format: 16-bit grayscale PNG, Bayer RGGB mosaic
  • Size: 256×256 pixels (1 channel raw)
  • Bit depth: 0–4095 (12-bit sensor in 16-bit container)
  • After de-Bayer: 4×128×128 float32 (R, G1, G2, B channels)
  • Normalization: ÷ 4095 → range [0, 1]

Ground Truth: Fujifilm RGB (fujifilm/)

  • Format: 8-bit RGB PNG
  • Size: 256×256 pixels, 3 channels
  • Bit depth: 0–255 per channel
  • Normalization: ÷ 255 → range [0, 1]
  • Purpose: Fujifilm color science = the learning target
De-Bayer: How a 256×256 RAW Becomes 4×128×128

A camera sensor captures one color per pixel (the Bayer pattern). Each 2×2 block has R, G, G, B. We extract each color into its own channel:

# Bayer pattern (each cell = 1 pixel on the sensor): R G1 R G1 R G1 ... <- rows 0, 2, 4 ... G2 B G2 B G2 B ... <- rows 1, 3, 5 ... # De-Bayer: split 256×256 into 4 channels of 128×128 raw = np.array(Image.open(png_path)) # (256, 256) uint16 h, w = raw.shape # 256, 256 rggb = raw.reshape(h//2, 2, w//2, 2) # (128, 2, 128, 2) rggb = rggb.transpose([1, 3, 0, 2]) # (2, 2, 128, 128) rggb = rggb.reshape([-1, h//2, w//2]) # (4, 128, 128) ← R, G1, G2, B rggb = rggb.astype(np.float32) / 4095.0 # normalize to [0, 1] rggb.tofile("input.bin") # 4×128×128×4 bytes = 262,144 bytes

QuadraticConnectionUnit (QCU) — Core Block

Quadratic activation: out = block1(x) × block2(x) + bias

Each QCU contains two ConvRep branches. Their outputs are combined by element-wise multiplication, not addition. This is the "quadratic" part — the network can model interactions between features that a purely linear (additive) block cannot. The bias added after the multiply is a learned per-channel offset.

Why 10 Branches Help Training

More gradient paths → better optimization

During backprop, each branch gets its own gradient signal. The 5×5 branch learns large context, 1×1 learns channel mixing, 3×1 and 1×3 learn directional edges, 3×3 learns mid-range patterns. All 10 push the model toward different minima simultaneously — the optimizer finds a better solution than any single branch alone. This is why New Module achieves +5.7 dB PSNR.

BatchNorm variants for regularization

Each plain branch has a BN twin. BatchNorm normalizes layer outputs during training → stabilizes learning, reduces internal covariate shift, and acts as a regularizer. At inference, BN is absorbed into the conv weights (γ, β, running_mean, running_var → adjust W and b). Zero inference overhead — but helps training significantly.

Squeeze-and-Excitation Attention (ATT Block)

Learns which color channels matter most — per image

Step 1 — Squeeze (GAP): Global Average Pool collapses each of the 12 channels to a single scalar → output shape (12,). Summarizes "how active is this channel overall?"


Step 2 — Excite (FC → PReLU → FC → Sigmoid): Two 1×1 convolutions with nonlinearity process the 12 scalars → output 12 attention weights in [0, 1].


Step 3 — Scale: Each channel's entire feature map is multiplied by its attention scalar. Channels encoding useful color info (e.g. green edges) get values near 1; noisy channels get suppressed toward 0.

TAIL — Sub-pixel Upsampling (PixelShuffle ×2)

(12, 128, 128) → (3, 256, 256) without interpolation

PixelShuffle rearranges channels into spatial dimensions: (C×r², H, W) → (C, r×H, r×W) where r=2. Takes 12 channels at 128×128 → rearranges into 3 channels at 256×256. Each 2×2 block in the output comes from 4 channels at the same spatial position in the input — no blurring, no interpolation artifacts. A final 3×3 conv refines the output, then clamp01 ensures valid [0,1] RGB.

Reparameterization — The Key Insight
Mathematical fusion of 10 training branches into 1 inference conv. Zero quality loss. 20× fewer parameters. This is what makes the CUDA C engine possible.
The Core Math

Any sum of linear operations on the same input = one linear operation

All branches receive the same input X. Each branch applies a convolution (linear operation). Their outputs are stacked and projected by conv_out (another linear operation):

# What happens during forward pass (10 branches): out = conv_out.weight @ stack([conv1(X), conv2(X), ..., conv10(X)]) # Since conv is linear: conv_out[:, i] @ conv_i(X) == (conv_out[:, i] ⊛ conv_i.weight)(X) # Therefore: sum over all branches = one fused conv: W_fused = sum_i(conv_out.weight[:, i, :, :] ⊛ conv_i.weight) b_fused = sum_i(conv_out.bias[i] * conv_i.bias + ...) # Fused conv produces IDENTICAL output to all 10 branches: out = F.conv2d(X, W_fused, b_fused) # PSNR vs 10-branch = 134 dB (bit-identical)
BatchNorm Absorption
# BN formula at inference: y = γ·(x − μ)/√(σ²+ε) + β # This is linear in x → absorb into conv: scale = gamma / sqrt(running_var + eps) W_fused = conv.weight * scale.reshape(-1, 1, 1, 1) # scale each output channel's filter b_fused = (conv.bias - running_mean) * scale + beta # shift the bias # Result: W_fused, b_fused replace both conv AND BN with zero extra cost at runtime

CUDA C Engine — syenet_isp.exe
A standalone CUDA binary. No Python, no PyTorch, no cuDNN. Loads 22.1 KB weights into on-chip constant memory, then runs 9 custom kernels per image.
Startup Sequence (runs once)
// 1. Load weights from .bin file into CPU memory FILE *fp = fopen(argv[1], "rb"); fread(weights_cpu, sizeof(float), 5640, fp); // 2. Copy to GPU constant memory (64KB on-chip cache) __constant__ float c_w[5640]; cudaMemcpyToSymbol(c_w, weights_cpu, 5640 * sizeof(float)); // Now ALL 9 kernels can read any weight in ~1 GPU clock cycle // 3. Pre-allocate GPU buffers ONCE (reused for all images) cudaMalloc(&d_input, 4*128*128 * sizeof(float)); // 256 KB — RAW input cudaMalloc(&d_head, 12*128*128 * sizeof(float)); // 768 KB — HEAD output cudaMalloc(&d_body, 12*128*128 * sizeof(float)); // 768 KB — BODY output cudaMalloc(&d_att, 12 * sizeof(float)); // 48 B — SE scalars cudaMalloc(&d_tail, 12*128*128 * sizeof(float)); // 768 KB — TAIL input cudaMalloc(&d_out, 3*256*256 * sizeof(float)); // 768 KB — RGB output
Per-Image Forward Pass
// Timer: device-side, no PCIe round-trip cudaEvent_t t0, t1; cudaEventRecord(t0, 0); // ── HEAD ────────────────────────────────────── conv2d<<<grid,block>>>(d_input, d_head, c_w + W_HEAD1, ...); // 5×5, 4→12ch prelu<<<...>>>(d_head, c_w + W_PRELU1, 12*128*128); conv2d<<<grid,block>>>(d_head, d_head, c_w + W_HEAD2, ...); // 3×3, 12→12ch prelu<<<...>>>(d_head, c_w + W_PRELU2, 12*128*128); conv2d<<<grid,block>>>(d_input, d_branch2, c_w + W_HEAD_B2, ...); // block2 mul_bias<<<...>>>(d_head, d_branch2, d_head, c_w + W_BIAS_H, ...); // QCU: ×+bias // ── BODY ────────────────────────────────────── conv2d<<<grid,block>>>(d_head, d_body, c_w + W_BODY1, ...); // 3×3, 12→12ch prelu<<<...>>>(d_body, c_w + W_PRELU3, 12*128*128); conv2d<<<grid,block>>>(d_head, d_body2, c_w + W_BODY2, ...); // 1×1, branch2 mul_bias<<<...>>>(d_body, d_body2, d_body, c_w + W_BIAS_B, ...); // ── ATTENTION ───────────────────────────────── gap<<<1, 12>>>(d_body, d_att, 128*128); // Global Average Pool fc1x1<<<1, 12>>>(d_att, d_att2, c_w + W_FC1, 12, 12); // FC layer prelu<<<...>>>(d_att2, c_w + W_PRELU4, 12); fc1x1<<<1, 12>>>(d_att2, d_att, c_w + W_FC2, 12, 12); sigmoid_ip<<<...>>>(d_att, 12); // → attention in [0,1] channel_scale<<<...>>>(d_body, d_att, 12*128*128, 128*128); // scale channels // ── TAIL ────────────────────────────────────── pixel_shuffle<<<grid,block>>>(d_body, d_tail, 128, 128, 12, 3, 2); // ×2 upsample conv2d<<<grid,block>>>(d_tail, d_out, c_w + W_TAIL, ...); // 3×3 refine clamp01<<<...>>>(d_out, 3*256*256); // → [0,1] cudaEventRecord(t1, 0); cudaEventSynchronize(t1); cudaEventElapsedTime(&ms, t0, t1); // pure GPU time, microsecond precision
All 9 Kernels Explained
Kernel 1
conv2d
One thread per output pixel. Each thread loops over all input channels and kernel elements (e.g. 4×5×5=100 multiplies for HEAD). Reads weights directly from __constant__ c_w[].
grid = (W_out, H_out, C_out)
block = (16, 16, 1)
Kernel 2
prelu
Parametric ReLU — learnable negative slope per channel. In-place. x = (x ≥ 0) ? x : alpha[channel] × x. Alpha comes from c_w[].
ceil(N/256) blocks
256 threads each
Kernel 3
mul_bias
QCU fusion: multiplies two branch feature maps element-wise, then adds per-channel bias. Fuses 3 operations (multiply, broadcast bias, add) into one kernel launch.
out = b1 × b2 + bias[c]
ceil(N/256) blocks
Kernel 4
gap (Global Avg Pool)
Collapses 12×128×128 feature map to 12 scalars for the SE attention block. Each thread handles one channel, using atomicAdd to accumulate 128×128 = 16,384 pixel values.
1 block, 12 threads
atomicAdd per channel
Kernel 5
fc1×1 (Fully Connected)
Implements the SE attention FC layers as 1×1 convolutions. Each output thread computes a dot product: out[j] = Σᵢ(in[i] × W[j×Cin+i]) + bias[j].
1 block, C_out threads
dot product per output
Kernel 6
sigmoid_ip
In-place sigmoid: x = 1/(1+exp(−x)). Converts SE FC2 raw outputs to attention weights in [0,1]. Uses CUDA's fast __expf() intrinsic.
x = 1/(1+__expf(-x))
ceil(N/256) blocks
Kernel 7
channel_scale
Applies SE attention: multiplies every pixel in channel c by att[c]. Useful channels (green, edges) get values near 1; noisy channels get suppressed toward 0.
data[c·HW+s] *= att[c]
ceil(N/256) blocks
Kernel 8
pixel_shuffle
Sub-pixel upsample ×2: rearranges (12,128,128) tensor into (3,256,256) by moving channel dimensions into spatial positions. No interpolation, no blur.
grid=(Wout,Hout,Cout)
block=(16,16,1)
Kernel 9
clamp01
Clamps all output RGB values to valid range [0,1] before writing to output buffer. Handles overflow from the final 3×3 tail conv.
x = max(0, min(1, x))
ceil(N/256) blocks

GPU Memory Hierarchy
Understanding why the 22.1 KB weights fit in __constant__ memory — and why that single decision drives most of the 6× speedup over PyTorch.
Memory Tiers — Speed vs Capacity
__constant__ cache (on-chip)
64 KB per GPU · ~1 clock cycle · broadcast to all threads in a warp
Dedicated read-only cache. When all threads in a warp read the same weight (e.g. a kernel coefficient), the hardware broadcasts it in a single cycle — no per-thread fetch. SYENet weights = 22.1 KB → fits entirely. Every conv kernel reads all its weights from here.
L1 / Shared Memory (on-chip)
32–48 KB per SM · ~4 cycles · per thread block only
The conv2d kernel uses shared memory to cache the input tile being processed by a 16×16 thread block. Instead of each of the 256 threads reading the same input pixel from global memory 9 times (for a 3×3 kernel), the tile is loaded once into shared memory and read 9 times locally.
L2 Cache (on-chip)
4 MB on Quadro T1000 · ~20–30 cycles · GPU-wide
Activation tensors that pass between kernels (e.g. d_head written by conv2d, then read by prelu) often stay in L2. Kernels that run back-to-back on the same buffer benefit from L2 reuse without explicit synchronization.
VRAM — GDDR6 (off-chip)
4 GB total · ~300–500 cycles · all SMs share bandwidth
Where d_input, d_head, d_body, d_out live. Each kernel reads its input activations and writes output here. This is the main bandwidth bottleneck — the reason adding more branches doesn't proportionally slow down the GPU (memory bandwidth, not FLOP count, is the limit).
PCIe x16 (host↔device)
~16 GB/s · thousands of cycles · CPU ↔ GPU transfer
Only used twice per image: upload input.bin (256 KB) and download output.bin (768 KB). The cudaEventRecord timer starts AFTER the upload and ends BEFORE the download — latency measurement is pure GPU compute only.

The __constant__ Memory Effect in Detail

PyTorch slim: weights in VRAM

When a conv2d kernel runs in PyTorch, it calls cuDNN's cudnnConvolutionForward. The weight tensor lives in VRAM global memory. First access: cache miss → ~400 cycle penalty. Subsequent accesses within the same kernel call: may hit L2 (~20 cycles). But every forward pass starts cold again from PyTorch's perspective because it doesn't manage which weights stay in L2 between calls.

CUDA C: weights in __constant__

Weights loaded once at startup via cudaMemcpyToSymbol. They stay in the 64KB constant cache for the entire lifetime of the program. Every image, every kernel, reads them at 1 cycle. More importantly: when all threads in a warp (32 threads) read the same weight value (e.g. the same kernel coefficient applied to different pixels), the hardware broadcasts it in a single read — 32 threads pay 1 cycle total, not 32 cycles.

The 6× summary: PyTorch spends ~80% of its time in framework overhead: Python GIL, dispatch layers, cuDNN algorithm selection, memory allocator. The GPU math itself is fast. CUDA C eliminates all the wrapper layers and reads weights from a cache that's ~400× faster than VRAM. The result: 106 FPS → 641 FPS on identical hardware with identical math.

CPU vs GPU — Why Slim Helps Differently
On CPU (TFLite): slim gives 9× speedup. On GPU (CUDA): both models are identical after .slim(). The reason reveals a fundamental difference between CPU and GPU compute models.
CPU Compute Model — Sequential, Compute-Bound

Original → Slim on CPU (TFLite/XNNPACK): 10 FPS → 92 FPS = 9× speedup

On CPU, branches run one after another (even with SIMD vectorization inside each branch). 10 branches = approximately 10× more multiply-accumulate operations (MACs). XNNPACK is highly optimized for individual convolutions, but cannot automatically parallelize 10 separate branch convolutions without explicit multi-threading overhead. So:


  • 110,984 params × 4 bytes = 443 KB — doesn't fit in L1 or L2 cache → cold cache every inference
  • Each branch = separate memory access pattern → cache thrashing across branches
  • 10× MACs + 10× memory reads → ~10× wall time
  • After slim: 5,640 params = 22 KB → fits in L2, single cache-warm conv, 9.2× faster
GPU Compute Model — Parallel, Memory-Bandwidth-Bound

Original → Slim on GPU: both end up at same architecture after .slim()

On GPU, the bottleneck is not how many MACs you do — it's how fast you can feed data to the cores. The GPU has 2,560 CUDA cores that all need to be kept busy simultaneously. For a 12×128×128 activation tensor:


  • 196,608 output pixels × 12 channels = 2.36M values to compute — keeps all cores busy regardless of branch count
  • DRAM bandwidth (GDDR6, ~200 GB/s) is the ceiling — adding more branches reads more weights but activations stay the same size → bandwidth doesn't increase proportionally
  • The GPU can overlap branch computation across SMs — even the original 10-branch model doesn't run branches strictly sequentially
  • After slim: same activation sizes, same bandwidth cost, same time. The speedup from slim on GPU comes from elsewhere (fewer params → smaller model file, better cache usage) not raw FLOP reduction

Summary: Three Different Speedup Mechanisms

CPU: reduce compute

10 branches → 1 branch = 10× fewer sequential operations. MACs are the bottleneck. Slim directly attacks the bottleneck.

GPU: eliminate framework

PyTorch overhead = 80% of wall time. CUDA C removes Python, dispatch, cuDNN. Framework elimination — not model size — is the lever.

GPU: on-chip weights

Moving 22.1 KB weights from VRAM (300 cycles) to __constant__ cache (1 cycle) removes the weight-read bottleneck for every forward pass, permanently.

The key insight: Reparameterization's main benefit is not the same on CPU and GPU. On CPU: fewer ops = fewer cycles. On GPU: the reparameterized 5,640 weights fit in __constant__ memory, and eliminating PyTorch's dispatch chain removes the real bottleneck. The 640 FPS is not because slim has fewer ops — it's because CUDA C runs those ops without any framework in the way.
